## 📁 核心文件位置与用途

### 根目录文件（案件级记忆）

#### 1. `[案件编号]案件信息.md`（主文件 - v2.1）
**位置**: `output/[案件编号]/[案件编号]案件信息.md`

**设计目的**：**案件管理看板** - AI优化的轻量级文件，快速加载和概览

**适用对象**：AI Agent、承办律师快速查阅

**内容结构**:
- 📋 案件基本信息（编号、案由、管辖法院等）
- 👥 当事人信息（原告/被告、联系方式）
- 💰 费用跟踪（律师费、诉讼费等）
- ⏰ 案件时间线（立案、送达、开庭、判决节点）
- ⚡ 近期重要任务清单（7天、待办、下一步）
- ⏳ 法定期限管理（应诉、举证、上诉期限）
- 👥 争议焦点与法律研究（当前争议、法条、案例）
- 💡 风险提示与注意事项（当前风险、策略建议）
- 🔄 更新历史（工作日志）

**特点**：
- ✅ 简洁高效，AI可快速加载和解析
- ✅ 提供案件全局视图
- ✅ 所有Agent的首要上下文来源
- ✅ 保持在根目录，访问便捷

**创建命令**:
```bash
# 使用 new-case 技能自动创建
"请为案件 [2025]京0105民初1234号 创建工作区"

# 或手动复制并重命名
cp ".claude/templates/case-templates/[案件编号]案件信息.md" \
   "output/[案件编号]/[2025]京0105民初1234号案件信息.md"
```

### 日程管理目录文件（详细记录）

#### 2. `00 - 📅 日程管理/[案件编号]工时记录.md`（详细记录 - v2.0）
**位置**: `output/[案件编号]/00 - 📅 日程管理/[案件编号]工时记录.md`

**设计目的**：**详细工时记录与费用核算** - 人工查阅的完整记录

**适用对象**：承办律师、财务核算、客户汇报

**内容结构**:
- 📊 本月工时统计（日期、时长、工作内容、费率、费用）
- 📁 本案文件清单（按目录分类的文件索引）
- 📈 案件进展摘要（月度关键进展、下一步计划、风险提示）
- 📊 累计工时统计（历史工时和费用汇总）
- 💡 重要说明与备注（工作范围、费用确认、特殊说明）

**特点**：
- ✅ 详细完整，支持审计查阅
- ✅ 按月度组织，便于费用结算
- ✅ 文件清单便于快速定位文档
- ✅ 放在日程管理目录，职责清晰
- ⚠️ 文件较大，AI不优先加载

---

#### 3. `00 - 📅 日程管理/[案件编号].yaml`（结构化数据）
**位置**: `output/[案件编号]/00 - 📅 日程管理/[案件编号].yaml`

**设计目的**：**结构化期限管理** - 机器可读的数据文件

**适用对象**：Scheduler Agent、自动化工具

**内容结构**:
- 案件基本信息（结构化字段）
- 法定期限数据（开始、截止、剩余天数）
- 任务清单（标题、日期、状态、负责人）
- 费用预算与支出
- 工作进度跟踪
- 风险预警指标

**特点**：
- ✅ 机器可读，支持自动化提醒
- ✅ 结构化数据，便于计算和分析
- ✅ Scheduler Agent的专用数据源
- ✅ 与工时记录.md形成互补

## 🎨 文档分层设计理念

### 为什么这样设计？

**第一层：根目录概览文件**
- **AI优化**：轻量级、快速加载、提供全局视图
- **决策支持**：AI快速了解案件全貌，做出智能决策
- **效率优先**：避免AI加载过多细节信息
- **统一入口**：所有Agent的上下文来源

**第二层：日程管理目录 - 详细记录**
- **人工查阅**：详细完整，支持审计和费用结算
- **职责清晰**：工时、期限、任务归口到日程管理
- **信息完整**：不遗漏任何细节，满足合规要求
- **按需加载**：仅在需要时读取，不影响AI性能

## 💡 使用场景

### 场景1：启动案件分析
1. AI读取 `案件信息.md` - 了解案件概况
2. 提取争议焦点、识别法律问题
3. 生成分析报告

### 场景2：生成法律文书
1. AI读取 `案件信息.md` - 获取当事人信息、案件事实
2. 引用法律研究、证据材料
3. 起草法律文书

### 场景3：费用结算
1. 人工打开 `工时记录.md`
2. 查看月度工时统计
3. 生成费用清单

### 场景4：期限提醒
1. Scheduler Agent读取 `.yaml` 文件
2. 检查法定期限
3. 发送提醒通知

## 🔄 数据流转

```
案件信息.md（根目录）
    ↑↓（双向同步）
    ├──→ 工时记录.md（详细数据）
    ├──→ yaml（结构化数据）
    └──→ 各Agent工作成果
```

**更新规则**：
- 初步信息 → 先更新案件信息.md
- 详细数据 → 补完工时记录.md
- 期限数据 → 更新 yaml
- 任务进展 → 三文件同步

## 📝 更新与维护

### 更新频率

- **案件信息.md**：每次Agent工作后立即更新
- **工时记录.md**：每天或每周集中更新
- **yaml**：实时更新（Scheduler自动维护）

### 维护责任

- **案件信息.md**：所有Agent共同维护
- **工时记录.md**：承办律师手动维护

### AI处理阶段

**启动阶段**:
1. 读取 `案件信息.md` 了解背景
2. 提取案件要素和争议焦点
3. 确定工作范围和策略

**处理阶段**:
1. 执行具体任务（分析、研究、起草）
2. 生成工作成果
3. 更新 `案件信息.md` 记录关键结果
4. （如需要）记录工时至 yaml

**完成阶段**:
1. 生成摘要和报告
2. 更新案件信息.md的状态
3. 提醒律师查看/更新工时记录.md（如需要）

---

### 人工处理阶段

**日常记录**:
1. 每天结束时查看/更新 工时记录.md
2. 记录工作时长、工作内容、费用
3. 更新任务状态、文件清单
4. 记录重要电话、会议、函件

**周度总结**:
1. 查看本周工时和费用
2. 检查任务完成度
3. 更新案件进展摘要
4. 规划下周工作

**月度结算**:
1. 整理本月工时统计
2. 计算本月费用
3. 生成工时报表
4. 向客户发送账单

**关键信息同步**:
1. 从工时记录.md中提取关键进展
2. 同步到案件信息.md（保持简洁）
3. 通知AI Agent有更新
4. AI下次读取时获取最新信息

---

### 同步机制

**AI → 人工**:
- AI完成任务后，在案件信息.md中记录要点
- 如涉及人工补充，在"下一步计划"中标记
- 如产生工时，在yaml中记录并提醒人工确认

**人工 → AI**:
- 人工更新工时记录.md
- 关键进展手动同步到案件信息.md
- AI下次读取案件信息.md时获取更新
- 如需AI立即响应，发送通知
